# v3 Hybrid BT+RL 협동 NPC 연구 노트 (논문 작성 참고용)

*정리 시점: 2026-04-22 | 최종 실험 버전: v35*

---

## 1. 연구 동기

### 1.1 Pure RL 의 구조적 한계 (v26 실증)

v26 실험 (B안 관측 123차원, 역할별 독립 PPO, 커리큘럼 학습, 300,000 에피소드):

| 지표 | v26 결과 |
|------|---------|
| 최종 승률 | **0%** (최근 5,000 ep 연속 wipe) |
| Phase 2 도달률 | 99.0% |
| Phase 3 도달률 | 17.4% 피크 → 10.1% 역진 |
| Stagger 기믹 성공 | **0회 / 5,000 ep** |

**원인 3가지**:
1. **희소 기믹 이벤트**: SEAL_BREAK · STAGGER 등은 페이즈 전이 시점에 1~2회만 발생 → 랜덤 탐색으로 성공 경험 누적 불가능
2. **Multi-agent credit assignment**: "누가 덕에 스태거 성공?" 개별 귀속 어려움
3. **엔트로피 붕괴**: 160k ep 이후 정책 deterministic → local minimum 고착 → 역진

→ **각 문제 유형에 적합한 패러다임을 할당하는 구조적 해법 필요**.

## 2. 제안: 2계층 하이브리드 아키텍처

### 2.1 설계 원리

**"이산·규칙 명확 = BT, 연속·맥락 의존 = RL"**

| 문제 유형 | 성격 | 적합 패러다임 |
|----------|------|-------------|
| 기믹 인식 대응 | 이벤트 이산, 규칙 명확, 실패 시 치명적 | **Behavior Tree** |
| 전투 포지셔닝·어그로·힐 | 연속 공간, 문맥 의존, 점진적 최적화 | **Reinforcement Learning** |

### 2.2 아키텍처

```
[관찰 obs, 123 dim]
    ↓
Layer 1: 기믹 인식 BT (결정론적 규칙, 6 우선순위)
    ↓ (규칙 fire 없으면 fall-through)
Layer 2: 역할별 RL (TankNet / HealerNet / SupportNet)
    ↓
[액션 17개 중 선택]
```

### 2.3 BT 규칙 우선순위 (6단계)

실험으로 검증된 최종 순서:

1. **SEAL_BREAK** (페이즈 전이, 최우선)
2. **STAGGER** (창 열리면 전원 딜 집중 — MARK/CHAIN/CROSS 무시)
3. MARK (표식자 분리)
4. CURSED_CHAIN (파트너 접근)
5. CROSS_INFERNO (안전 사분면 이동)
6. 임박 위험 (≤2턴 탈출)

**핵심 결정**: STAGGER 를 MARK/CHAIN 보다 우선 → 스태거 타임은 모든 것 포기하고 딜 몰빵 (MMORPG 관례 정합).

## 3. 관찰 공간 재설계 (v26 B안, 123 dim)

### 3.1 이전 구조의 문제

기존 텔레그래프 블록은 "슬롯 2개 × 각 슬롯 17 dim" → **슬롯 순서 비결정적** (같은 SLASH 가 때로 슬롯 0, 때로 슬롯 1).

### 3.2 B안 개선

- **패턴 ID 고정 채널** (9 패턴 × 5 dim = 45 dim): active, turns_norm, am_I_target, target_dx, target_dy
- **SEAL NPC spot 배정 알고리즘**: 헝가리안 그리디 매칭 → 각 NPC 에게 고유 spot 지정
- **협동 컨텍스트 블록** (13 dim): Chain/Cross/Stagger/Seal 전용 정보

결과 구성:
```
Self(15) + Allies(24) + Boss(10) + PatternChannels(45) +
Danger(8) + Escape(4) + Coop(13) + Player(4) = 123
```

## 4. 환경 설계 핵심 결정 (v30~v35)

### 4.1 물리 규칙

**파티원 간 충돌 제거, 보스 충돌만 유지**
- 이전: 모든 유닛 간 원형 충돌 → 스태거 집결 시 탱커가 딜러에 막혀 못 붙음
- 수정: 파티원 통과 가능, 보스만 몸통 차단 (MMORPG 관례)

### 4.2 보스 행동 (스태거 wind_up 6턴)

두 가지 제약 추가:
1. **보스 이동 정지** — 파티가 집결할 시간 확보
2. **신규 패턴 트리거 차단** — 파티 흩어짐 방지

### 4.3 밸런스 튜닝 (v33)

**전원 HP × 2 (데미지·방어·공격값 고정)**:

| 유닛 | HP (v26) | HP (v33+) |
|------|----------|-----------|
| Boss | 900 | 1800 |
| Tank | 340 | 680 |
| Healer | 140 | **280** |
| Dealer | 170 | 340 |
| Support | 180 | 360 |

**근거**: 힐러 140 HP 는 Stagger fail (115 net dmg) 한 번에 반피, 두 번에 사망. HP ×2 로 **discrete burst damage** 에 대한 버퍼 확보 → 연쇄 wipe 방지.

### 4.4 에피소드 길이 (v35)

`max_steps: 300 → 1000` — 보스 HP 증가와 연동하여 타임아웃 소멸.

## 5. 학습 파이프라인

### 5.1 하이브리드 통합 (train_boss.py)

```python
for uid in NPC_slots:
    bt_action = BTPolicy.act()
    if bt_action is not None:
        use bt_action  # RL buffer 저장 X
    else:
        use RL network  # RL buffer 저장 O → 학습
```

**효과**: RL 은 "기믹 없는 순수 전투" 만 학습 → 학습 공간 대폭 축소 → 수렴 가속.

### 5.2 PPO 설정

- 역할별 독립 네트워크 (TankNet / HealerNet / SupportNet)
- 엔트로피 계수: 0.08 (v26 의 0.04 에서 상향 — 엔트로피 붕괴 방지)
- 버퍼 크기: 1024 (v26 의 256 에서 확대)
- 딜러 슬롯: FSM 정책 고정 (플레이어 자리)

### 5.3 보상 설계 (v26 B안 재활용)

- 계층적 3단: 역할 수행 < 패턴 회피 < 보스 처치
- MARK/CROSS 그라디언트 shaping (이분법 제거)
- SEAL 배정 spot 기반 shaping
- Stagger contribute 이벤트 반영

## 6. 핵심 실험 결과

### 6.1 버전별 승률 추이

| 버전 | 학습 에피소드 | 승률 | Stagger 성공률 | Mechanic 성공률 | P3 도달률 |
|------|-------------|------|--------------|---------------|---------|
| v26 (Pure RL baseline) | 300,000 | 0% | 0% | ~45% | 10% |
| v32 (하이브리드 최초 fresh) | 294 | 62% | 6.2% | 94.4% | 70% |
| v33 (HP ×2) | 216 | 56% | 43.9% | 77.8% | 75% |
| **v35 (최종)** | 244 | **56%** | **37.1%** | **82%** | **70%+** |

### 6.2 최종 개선 폭 (v26 → v35)

| 지표 | 개선 |
|------|------|
| 승률 | 0% → 56% (**+56%p**) |
| Stagger 성공 | 0% → 37% (**+37%p**) |
| Mechanic 성공 | 45% → 82% (+37%p) |
| P3 도달 | 10% → 70% (+60%p) |
| 학습 시간 | 17.6 시간 → 수 분 안 50%+ 도달 |

### 6.3 Peak 순간 관찰 (v35)

- ep 121-140: 승률 85%, 보스 딜 89%, P3 도달 85%
- ep 181-200: 승률 75%, 보스 딜 89%, P3 도달 85%
- 정책 수렴 가능성 확인, 변동폭만 축소 필요

## 7. 논문 작성 시 주요 포인트

### 7.1 Novelty 포지셔닝 (정직)

**이미 존재하는 것**:
- BT+RL hybrid 개념: AMD Schola (2025), MARL-BT (2023)
- MMORPG 레이드 RL 환경: RaidEnv (2023)
- Hierarchical MARL: HAVEN (AAAI 2022) 등

**본 연구의 좁은 차별점**:
1. **텔레그래프 기반 협동 파훼 기믹 특화** (SEAL/MARK/STAGGER/CHAIN/CROSS 8종)
2. **파티 측 학습** (UPC WoW 논문은 보스 측)
3. **Pure RL 수렴 실패 300k ep 정량 실증** (후속 연구 참조 자료)
4. **역할별 독립 네트워크 + 커리큘럼 + BT 하이브리드의 누적적 통합**

### 7.2 연구 라인 포지셔닝

| 단계 | 질문 | 기여 |
|------|------|------|
| v1 (선행 연구) | 대칭 5v5 협동 가능? | 파라미터 공유 + 셀프플레이 |
| v2 (paper_draft) | 역할 이질성? | 역할별 독립 네트워크 |
| v3 (본 연구) | 희소 기믹? | BT 인식 + RL 실행 하이브리드 |

각 단계가 앞 단계의 구체적 한계를 해결. v3 는 누적적 발견의 종착점.

### 7.3 공헌 포인트

1. **실패 사례 실증**: Pure RL 로 희소 기믹 학습 불가 — 정량 증거 (v26)
2. **구조적 분해 설계**: 이벤트 유형 × 규칙 명확성 판정 기반 BT/RL 분배
3. **구체 구현 레퍼런스**: BT 6 규칙, Layer 2 학습 통합 코드
4. **환경 설계 원칙**: HP 스케일, 스태거 중 보스 정지, 파티원 통과 등

### 7.4 한계 / Future Work

1. **Variance**: v35 에서도 50 ep 단위 승률 변동 ±20%p — 장기 학습 또는 앙상블로 축소 필요
2. **Stagger 40% 한계**: 2인 이상 사망 시 수학적 불가능 영역 존재
3. **분해 판정 기준의 novelty**: "이벤트 희소성 × 규칙 명확성" framework 이 완전 신규인지 저널 투고 전 추가 문헌 조사 필요
4. **사용자 평가 미실시**: Phase 3 (GEQ, Trust, NASA-TLX) 진행 예정

## 10. 끝으로

핵심 문제는 **"RL 단독으로는 4인 협동 희소 기믹을 학습할 수 없다"** 이고, 해법은 **"기믹 인식은 규칙, 전술 실행은 학습"** 의 원칙적 분해였다. v26 → v35 과정에서 발견한 6개 설계 결정 (관찰 재구성, BT 우선순위, 파티 충돌 제거, 보스 정지, HP 스케일, 에피소드 길이) 이 논문의 실질적 공헌이 된다.
